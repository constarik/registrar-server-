# 3B — Firestore-персист розыгрышей (спека на ревью, 2026-06-13)

Чинит аудит-находку **B2**: сейчас pending-сессии лежат в `STATE_DIR` (tmpdir), который на Render эфемерен — рестарт в окне commit→reveal стирает `serverSeed`, и розыгрыш невосстановим. Firestore делает состояние durable; попутно открывает публичное чтение результата без оператора и безопасные отложенные розыгрыши (5-минутное окно становится не страшным).

## ГЛАВНАЯ МИНА — секретность `serverSeed`

`serverSeed` **секретен до reveal** и **публичен после** (он намеренно отдаётся в ответе `/reveal` — это и есть раскрытие). Значит durable-хранилище обязано разделять две фазы:

- **до reveal:** `serverSeed` лежит в Firestore, но **не должен** читаться никем, кроме сервера. Иначе любой вычислит результат заранее → грайндинг/инсайд.
- **после reveal:** `serverSeed` уже публичен, запись можно отдавать всем.

**Решение — две коллекции:**
- `uvs_draws_pending` — приватная. Хранит `serverSeed`, `anchor`, `commitmentHash` и пр. на время окна. Пишет/читает только сервер через admin SDK.
- `uvs_draws` — публичная (как `paddla_games`). Пишется ТОЛЬКО в момент reveal, содержит полную раскрытую запись (где `serverSeed` уже легально публичен). Это и durable-результат, и публичный trail, и ответ на «опубликовать без меня».

**Статус правил Firestore (проверено 2026-06-13): мина уже обезврежена, правки НЕ нужны.**
Текущие правила заканчиваются `match /{document=**} { allow read, write: if false; }`, а единственный вайлдкард (`{collection}/{doc}`) ограничен `collection.matches('kolumn_daily_.*')`. Значит любая коллекция `uvs_*` по умолчанию закрыта для всех клиентов; admin SDK (которым пишет регистратор) правила обходит. `serverSeed` в `uvs_draws_pending` физически недоступен клиентам без единой правки.
**Два условия к коду (оба тривиальны):**
1. Имена коллекций — вне `paddla_*` и `kolumn_daily_*`. `uvs_draws_pending` / `uvs_draws` подходят.
2. Публичное чтение раскрытых розыгрышей — ТОЛЬКО через HTTP API регистратора (`GET /draws/:id`, admin SDK), НЕ через прямой клиентский read Firestore. Поэтому `uvs_draws` тоже остаётся закрытой в правилах — наружу её отдаёт код, а не Firestore.

## Изменения

**И1 — server.js (1 строка):** передать существующий хэндл в модуль:
`mountAnchoredDraws(app, { trailDb, trailEnabled })` вместо `mountAnchoredDraws(app)`.
Это нарушает прежнее «только anchored-draws.js», но иначе модуль не получит Firestore. Минимально и неизбежно.

**И2 — store-абстракция (anchored-draws.js):** ввести `store` с async `{ putPending, getPending, delPending, listPending, putРевealedPublic }`.
- Если `trailDb` есть → Firestore (`uvs_draws_pending` / `uvs_draws`).
- Если нет → текущие файлы в `STATE_DIR` (полное сохранение нынешнего поведения, нулевая регрессия для локального теста и для случая «Firebase недоступен»).
Текущий объект `pending` (sync, файловый) становится файловым бэкендом этого store.

**И3 — async-переход:** `/commit`, `performReveal`, тикер уже `async` — заменяю sync-вызовы `pending.*` на `await store.*`. Риск: не забыть `await` нигде (иначе гонки/потерянные записи). Аудит сосредоточить здесь.

**И4 — reveal пишет в публичную коллекцию:** в конце `performReveal`, после сборки `record`, — `await store.putRevealedPublic(drawId, record)` и `delPending(sessionId)`. Идемпотентность сохраняется: повторный reveal видит, что запись уже в `uvs_draws`, и отдаёт её.

**И5 — публичное чтение:** `GET /draws` (список последних) и `GET /draws/:drawId` (одна запись) из `uvs_draws`. Так розыгрыш виден всем и после твоего ухода, и `/draw-status` может читать durable-статус.

## Что НЕ делаю в этом патче
- Отложенные розыгрыши (`delaySeconds`) — отдельно, ПОСЛЕ Firestore (теперь они станут безопасны по durability). Не глотать вместе.
- UI страницы `/draw` под публичное чтение — отдельно, фронтенд.
- Миграцию `paddla_games` — не трогаю, чужая коллекция.
- Крипту, anchor-верификацию, derived-R, TSA — руки прочь.

## Риски для аудита
1. **Утечка `serverSeed`** через правила Firestore — главный; закрывается приватной коллекцией + правилом запрета read (вне кода).
2. **Пропущенный `await`** при sync→async переходе — гонки/потеря записи. Тест: commit → проверить запись в pending до reveal → reveal → проверить переезд в public + удаление из pending.
3. **Деградация без Firebase** — store обязан падать на файлы, не на исключение. Тест: прогнать весь сценарий с `trailDb=null` (как сейчас) → 9/9 прежних проверок должны пройти.
4. **Двойной reveal в гонке** (как в тикере) — Firestore `.set()` идемпотентен по `drawId`; результат детерминирован → last-write-wins безвреден, как и в файловой версии.
