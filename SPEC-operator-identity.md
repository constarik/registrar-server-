# uvLottery — operator identity (ЧЕРНОВИК спеки, 2026-06-13)

*Статус: draft на обсуждение, НЕ к реализации. Цель — зафиксировать модель, прежде чем трогать код или стандарт.*

## Задача

Один сервис (registrar) — много независимых операторов розыгрышей. Нужно, чтобы любой третий мог **неподделываемо** установить, КТО провёл розыгрыш — при том что:
- операторы могут называться одинаково или почти (омоглифы: кириллическая «о», табуляция вместо пробела, zero-width пробел);
- сервис сам недоверяем (он лишь хост);
- имя задаётся свободным текстом в `/commit` и потому ничего не доказывает.

Это новый **необязательный** слой uvLottery. Розыгрыш без идентичности остаётся валидным (как сейчас) — просто «оператор не заявлен/не доказан».

## Принцип

**Личность оператора = публичный ключ, а не имя.** Имя — недоверяемый ярлык для людей; проверка идёт только по ключу. Это та же ось, что и весь UVS («проверяй математику, не надпись») и что §5.4 («trust anchor не от оператора»).

## Криптопримитив

- Пара ключей **ed25519** у оператора. Приватный НИКОГДА не покидает оператора (как ephemeral key игрока в uvGs §7).
- В `/commit` оператор подписывает `commitmentHash`:
  `operatorSig = ed25519_sign(operatorPrivKey, commitmentHash)`
- Запись несёт: `operator: { pubKey, sig, name? }`.
- RFC-3161-штамп уже доказывает *что* и *когда*; подпись добавляет *кто*. Вместе: «ключ K подписал ровно этот розыгрыш до раунда R».

## Что доказывает подпись и что нет

- Доказывает: «розыгрыш с этим `commitmentHash` подписан владельцем приватного ключа к `pubKey`».
- НЕ доказывает сам по себе: «pubKey принадлежит настоящей Org A». Это устанавливается **вне записи** — проверяющий берёт ключ Org A из её официального канала (см. «Распределение ключей»). Запись лишь самосогласованна; привязка ключ→организация — отдельный, операторский слой доверия.
- Омоглиф-атака растворяется: совпадение имён ничего не даёт без приватного ключа; «Org A»(K1) и «Org A\t»(K2) — два разных оператора по построению.

## РЕШЕНИЯ К ОБСУЖДЕНИЮ (заполнить до FAS)

### Р1. Распределение ключей — как проверяющий узнаёт «настоящий» pubKey оператора?
- **(а) Self-sovereign (рекоменд. для MVP):** оператор публикует свой pubKey в своём официальном канале — DNS TXT на домене, `/.well-known/uvs-operator.json`, или просто на сайте/в соцсетях. Реестра нет, доверяемой третьей стороны нет. Якорь = опубликованный ключ. Максимально в духе UVS.
- **(б) Нейтральный реестр имя→ключ через регистрацию+2FA:** сервис ведёт справочник; оператор регистрируется, проходит **доменную верификацию** (доказывает контроль над orgA.com), и это связывает имя→ключ; **2FA (TOTP/passkey)** защищает саму регистрацию от захвата. Удобнее для людей, но реестр становится доверяемым и обязан разруливать коллизии имён. Прецедент: core §10.1 «neutral registry key», uvGs §5.3 WHITELIST. ВАЖНО: реестр дополняет подпись, не заменяет — проверяемый артефакт всё равно подпись против ключа.
- **(в) Гибрид:** self-sovereign как база + опциональная регистрация (б) для «проверенных».
- *Черновой выбор: (а), с (в)/(б) как добавкой позже.*

> **2FA: что решает и чего НЕ решает (зафиксировать, чтобы не подорвать модель).**
> 2FA — аутентификация (кто говорит с сервисом СЕЙЧАС); стандарту нужна атрибуция, проверяемая третьим лицом БЕЗ доверия к сервису (кто провёл розыгрыш, доказуемо офлайн). Разные слои:
> - 2FA защищает вход/регистрацию аккаунта. Но «розыгрыш провёл аккаунт Org A» — утверждение БАЗЫ сервиса; взломанный/злонамеренный сервис его подделает → возвращается доверенный посредник, которого UVS убирает.
> - Подпись ed25519 проверяется офлайн против опубликованного ключа; взлом сервиса её не подделает (приватного ключа у сервиса нет).
>
> **Линия, которую нельзя переходить:** проверяемый артефакт атрибуции — ВСЕГДА подпись против опубликованного ключа. Регистрация/2FA — слой удобства/кастодии/анти-спама ПОВЕРХ подписи, никогда вместо. Если «кто провёл» сведётся к «залогинься, и мы поручимся» — стандарт теряет смысл и противоречит своему питчу («не надо доверять оператору или сервису»).
>
> **Три задачи, которые 2FA-регистрация решает легитимно:** (1) кастодия ключа для нетехнических операторов (сервис хранит ключ, отпирает по 2FA, подписывает от их имени — см. custody-tier ниже); (2) реализация реестра Р1(б) с доменной верификацией; (3) защита публичного `/commit` от спама.

### Р1а. Custody tier — тирить кастодию, не запрещать (применяем 🟢/🟡/🔴-принцип к личности)
Запись честно сообщает режим подписи, проверяющий решает сам:
- **self-custodied** — приватный ключ держит оператор, подпись офлайн-проверяема, сервис ключа не видел → сильнейший;
- **service-custodied + 2FA** — ключ под кастодией сервиса, отпирается 2FA; удобнее, но сервис теоретически мог подписать сам → слабее;
- **unsigned** — оператор не заявлен (`operator: null`, как сейчас) → слабейший.

Тир личности — ОТДЕЛЬНАЯ ось от тира розыгрыша (🟢 outcome-binding). Розыгрыш бывает 🟢 по честности и при этом self-/service-custodied/unsigned по личности. Не смешивать две оси в одном бейдже.

### Р2. Что именно подписывается?
- Кандидат: `commitmentHash` (через него — участники, пул, commitment, chainHash). Плюс: уже привязан к TSA-времени.
- Открыто: включать ли в подписанные данные `operator.name`, чтобы имя тоже было под подписью? Если да — имя нельзя подменить постфактум, но всё равно нельзя верить (омоглиф под валидной подписью возможен у самого же оператора). Скорее: подписывать `canonicalJSON({ commitmentHash, operatorName })`, чтобы имя было зафиксировано оператором, не сервисом.
- *Черновой выбор: подписывать `canonicalJSON({ commitmentHash, name })`.*

### Р3. Отображение (анти-омоглиф на уровне UI/списка)
- Истинный id в списке = короткий **отпечаток ключа** (например, первые 16 hex от `SHA-256(pubKey)`), не имя.
- Имя показывать NFC-нормализованным, помеченным `claimed`, и **флажить подозрительное**: непечатаемые/zero-width символы, смешение скриптов (латиница+кириллица), ведущие/хвостовые пробелы. Не запрещать — помечать (⚠), потому что запрет — это политика, а UVS показывает факты.
- Индикатор «verified»: если проверяющий передал ожидаемый pubKey (из офиц. канала оператора) — список/запись показывает «✓ подпись совпала с ожидаемым ключом» против «claimed, ключ не сверялся».

### Р4. Роль сервиса
- Сервис **только носит** `operator.{pubKey,sig,name}` и проверяет, что подпись валидна (иначе reject на `/commit` — мусорную подпись не хранить). Сам сервис НЕ ручается за принадлежность ключа.
- Опц. WHITELIST (Р1в): сервис помечает розыгрыши от известных ему ключей. Прецедент uvGs §5.3 — «whitelist не мешает копировать код, но мешает копии пройти как verified».

### Р5. Совместимость
- Слой необязательный: `/commit` без `operator` → розыгрыш как сейчас, поле `operator: null`. Существующие записи не ломаются.
- В `uvLs.md` — новая секция «Operator identity (optional)», по образцу uvGs §7 (тот же ed25519-примитив, перенесённый с игрока на оператора).

## Объём реализации (после FAS, ОТДЕЛЬНО)
1. `uvLs.md` — нормативная секция (сначала спека стандарта, потом код — твой же порядок).
2. Верификаторы (js/py/java/c++) — опц. проверка `operatorSig` (stdlib ed25519 есть в Node `crypto`, Python `cryptography`, Java `java.security`, C++ — сложнее, возможно отметить как «verifier-optional»).
3. `anchored-draws.js` — `/commit` принимает `operator:{pubKey,sig,name}`, валидирует подпись, кладёт в запись; `/draws` показывает отпечаток+claimed-имя+флаги.
4. `/draw`-страница — генерация ключа в браузере (или ввод), подпись commitment.

## Риски для будущего аудита
1. **Привязка ключ→организация** — самое слабое звено, и оно ВНЕ крипто (как input-honesty). Честно сказать в спеке: UVS доказывает «этот ключ подписал», не «этот ключ — это Org A». Последнее — на офиц. канале оператора.
2. **Потеря/компрометация приватного ключа оператора** — нужна ротация/отзыв? Для MVP: ключ на розыгрыш или долгоживущий ключ оператора? Долгоживущий удобнее для узнавания, но компрометация дороже. Открытый вопрос.
3. **C++-верификатор и ed25519** без внешних либ — может не влезть в «stdlib-only»; тогда identity-проверка = «verifier-optional», а draw-проверка остаётся во всех четырёх. Не смешивать обязательность.
4. **Подпись покрывает имя, но имя всё равно недоверяемо** — не создать у пользователя ложного ощущения, что «подписанное имя = настоящее». UI должен говорить «подпись валидна» отдельно от «это правда Org A».
